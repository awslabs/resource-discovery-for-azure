# Design Spec — Master HTML Summary (aggregate report across subscriptions)

Status: DRAFT / design-only (no code in this branch)
Author: RDA maintainers
Related: `Extension/Summary.ps1` (per-subscription HTML), `Run-AllSubscriptions.ps1`
(multi-sub wrapper), `Run-AllSubscriptions.Stream.ps1` (parallel streaming variant)

---

## 1. Problem

`Run-AllSubscriptions.ps1` produces one self-contained HTML report per
subscription (via `Extension/Summary.ps1`). On a tenant with many
subscriptions the operator ends up with a folder of dozens — or hundreds — of
per-subscription HTML files. To answer a simple question such as "did the run
actually cover everything, and roughly how much is in each subscription?" the
operator has to open each file one at a time. There is no single place that
rolls the whole run up.

This spec proposes a **master HTML** — a self-contained "HTML of HTMLs" that
aggregates every per-subscription report from a single wrapper run into one
page: a run-wide totals view, a per-subscription table, and links out to the
individual reports. It is a verification aid first (did we cover every sub,
are any empty/failed, how do the totals distribute) and a navigation aid
second (jump to any per-sub report).

## 2. Goals / Non-goals

### Goals
- One self-contained `.html` (no CDN/JS libs), same portability contract as
  `Extension/Summary.ps1` output — openable from a file share, Cloud Shell, or
  email attachment.
- Run-wide totals: total resources, total subscriptions processed, empty subs,
  failed subs, and a by-service aggregate across the whole run.
- Per-subscription row view: subscription name, resource count, and a link to
  that subscription's own HTML report.
- Surface the run-health signals the wrapper already tracks: failed
  subscriptions, consumption-failed subs, metrics-failed subs, collector
  failures (see §5).
- Honour obfuscation: when the run was obfuscated, the master must not reveal
  anything the per-sub reports masked (see §6).
- Show a progress bar while it builds, using the shared `Write-RdaProgress`
  helper (see §8).

### Non-goals
- No cross-subscription de-duplication of individual resources (each sub's
  resources are counted independently; a resource can only live in one sub
  anyway).
- No re-querying Azure. The master is built purely from artefacts already on
  disk from the wrapper run. It never makes ARM/Graph calls.
- No replacement of the per-sub reports. The master links to them; it does not
  inline their full per-service tables.
- Not a historical/trend tool. One master = one wrapper run.

## 3. Where it fits

The master is generated by the multi-subscription wrappers **after** all
per-subscription runs finish, at the point where the wrapper already has:
- the list of subscriptions it attempted,
- `$SubResourceCounts` (a list of `{ Name, Id, Count }` — see
  `Run-AllSubscriptions.ps1`, populated in both the sequential and the parallel
  stream-aggregation paths),
- the run-health collections (`$FailedSubscriptions`,
  `$Global:ConsumptionFailedSubs`, `$Global:MetricsFailedSubs`,
  `$Global:CollectorFailures`),
- the output directory holding each subscription's report folder / zip.

It is invoked the same way the other extensions are — `& $MasterSummaryPath ...`
— so it inherits the wrapper's scope and needs no new global state.

## 4. Data sources — two tiers

The master supports two levels of richness. Tier 1 is cheap and always
available; Tier 2 is richer but requires parsing each per-sub inventory.

### Tier 1 — lightweight index (default, no per-sub parsing)
Built entirely from data the wrapper already holds in memory:
- Per-subscription **name + resource count** from `$SubResourceCounts`.
- Run totals = sum of counts; count of empty subs (Count 0); count of failed
  subs (from `$FailedSubscriptions`).
- Health banners from the `$Global:*FailedSubs` / `$Global:CollectorFailures`
  collections.
- A link per subscription to its per-sub HTML (resolve the path/filename the
  wrapper wrote for that sub).

Tier 1 needs **zero** file re-reading beyond discovering the per-sub HTML paths,
so it is fast even on a very large tenant and works even when only counts (not
full inventories) are retained.

### Tier 2 — rich per-service aggregate (opt-in, parses per-sub JSON)
For a by-service roll-up across the whole run, parse each subscription's
`Inventory_*.json` (the same file `Extension/Summary.ps1` consumes: top-level
keys are service-type names, values are record arrays). Accumulate a
`service -> total count` map across all subs, then render the same donut + bar
charts `Summary.ps1` already produces, but over run-wide totals.

Tier 2 is where the progress bar earns its place: on a large tenant this loop
opens and parses one JSON per subscription, which is the slow part (§8).

The tier is selected by a switch (e.g. `-Detailed`), defaulting to Tier 1 so
the master stays cheap unless the operator asks for the service breakdown.

## 5. Run-health integration

The wrapper already aggregates per-phase health across streams. The master
should surface these as banners / small tables so the "did we really cover
everything" question is answered on the same page:

| Source collection | Meaning | Master rendering |
|---|---|---|
| `$FailedSubscriptions` | subs whose run failed | red banner + list |
| `$SubResourceCounts` Count == 0 | subs that returned 0 resources | amber banner + list (often a Reader-permission gap) |
| `$Global:ConsumptionFailedSubs` | consumption phase auth issues | amber note (per-sub) |
| `$Global:MetricsFailedSubs` | metrics phase auth issues | amber note (per-sub) |
| `$Global:CollectorFailures` | per-(sub, collector) throws | amber note grouped by sub |

These are the approved health globals (see steering `anti-patterns.md`); the
master only READS them — it introduces no new globals.

## 6. Obfuscation handling

The master must never weaken the obfuscation posture of the run it summarises.

- **Counts are always safe.** Resource counts, sub counts, and per-service
  totals reveal no identity, so Tier 1 and the Tier 2 charts are safe in both
  obfuscated and identifiable runs.
- **Names are the sensitive part.** In an obfuscated run the subscription
  "name" available to the wrapper may itself be a token; render whatever the
  per-sub report rendered — do not attempt to resolve tokens back to real
  names (the master has no dictionary and must not load one).
- **Detect posture the same way `Summary.ps1` does** — reuse its obfuscation
  signature check (the `^(prod_|nonprod_)<guid>$` pattern sampling) so the
  master's privacy banner matches the per-sub reports. If posture is unclear,
  default to the safe "identifiable" label (same conservative default as
  `Summary.ps1`).
- Links to per-sub reports point at the already-written per-sub HTML files,
  which are themselves already obfuscated when `-Obfuscate` was used. The
  master adds no new disclosure surface.

## 7. Reuse of `Extension/Summary.ps1` helpers

`Summary.ps1` already contains self-contained, dependency-free building blocks
the master should reuse rather than reinvent:
- `ConvertTo-HtmlSafe` — edge HTML-escaping for every embedded string.
- `New-DonutChart` / `New-BarChart` — hand-rolled SVG charts (no JS libs).
- The inlined CSS/JS block (search + click-to-sort tables, print rules).
- The obfuscation-detection pattern and privacy-banner markup.

Two viable approaches for sharing:
1. **Extract the helpers** into `Functions/Common.Functions.ps1` (already the
   shared dot-source home for `Write-RdaProgress`) and have both `Summary.ps1`
   and the master dot-source them. Cleanest long-term; touches `Summary.ps1`
   (needs its own review + scenario-matrix pass since it is output-affecting).
2. **Copy the small helpers** into the master extension. Zero risk to the
   per-sub report, at the cost of a little duplication.

Recommendation: start with (2) for a self-contained first version, and only do
the (1) extraction as a separate, dedicated change if/when a third consumer
appears — keeps the master change from becoming an output-affecting edit to the
per-sub report generator.

## 8. Progress reporting

The Tier 2 loop (parse one `Inventory_*.json` per subscription) is exactly the
kind of bounded, countable loop `Write-RdaProgress` (in
`Functions/Common.Functions.ps1`, from the `feature/unified-progress` work) is
designed for:

```
$total = $subs.Count
$i = 0
foreach ($sub in $subs)
{
    $i++
    Write-RdaProgress -Activity 'Building master summary' `
        -CurrentItem $sub.Name -Index $i -Total $total
    # ... parse that sub's Inventory JSON, accumulate service totals ...
}
Write-RdaProgress -Activity 'Building master summary' -Completed
```

- Interactive host: a live bar `Building master summary: <sub> (i of N)`.
- Non-interactive host: falls back to a throttled line (or use `-BarOnly` if
  the master is ever built inside a stream worker — it is not today, so the
  default line is fine and useful).

Tier 1 is effectively instant and does not need a bar.

> Dependency note: the master's progress relies on `Write-RdaProgress`, which
> lands with the `feature/unified-progress` branch. This design branch is
> intentionally based off `origin/main` and carries no code; the eventual
> implementation branch should be based on (or rebased after) the branch that
> ships `Common.Functions.ps1`.

## 9. Proposed file & interface

New extension: `Extension/MasterSummary.ps1` (mirrors `Extension/Summary.ps1`).

```
param(
    # Directory containing the per-subscription report folders/zips from the run.
    $RunOutputDirectory,

    # Output path for the master .html.
    $HtmlFile,

    # Per-sub index: array of { Name; Id; Count; ReportPath } the wrapper builds
    # from $SubResourceCounts + the per-sub HTML locations.
    $SubIndex,

    # Optional run-health collections (arrays), passed through from the wrapper.
    $FailedSubscriptions,
    $ConsumptionFailedSubs,
    $MetricsFailedSubs,
    $CollectorFailures,

    # Display-only header fields.
    $TenantId,
    $Version,
    $PlatOS,

    # Tier 2 (per-service aggregate) when set; Tier 1 index-only when omitted.
    [switch]$Detailed
)
```

The wrapper builds `$SubIndex` (it already has every element) and calls
`& $MasterSummaryPath @args` once, after the per-sub loop, gated behind a
`-MasterSummary` switch on `Run-AllSubscriptions.ps1` so existing behaviour is
unchanged by default.

## 10. Output layout (the master page)

```
Header:  tenant, version, platform, generated timestamp, privacy banner
Banners: failed subs (red), empty subs (amber), phase-health notes (amber)
Totals:  [ total resources ] [ subscriptions processed ] [ empty ] [ failed ]
Charts:  (Detailed only) run-wide donut (by service) + top-N bar (by service)
Table:   one row per subscription:
         | Subscription | Resources | Health | Report |
         |  <name>      |   1,234   |  ok    | open ↗ |   (link = per-sub HTML)
Footer:  RDA version, generator note
```

The per-sub links use relative paths so the master travels with the report
folder (matching the self-contained/portable contract).

## 11. Edge cases & failure handling

- **A per-sub Inventory JSON is missing/unreadable (Tier 2).** Skip that sub's
  service accumulation, still render its Tier 1 row, and mark it in the health
  column. Never abort the whole master for one bad file (mirrors the tool's
  fail-soft posture for the trivial long tail, while the wrapper's loud health
  reporting already captured the real failure).
- **Zero subscriptions / empty index.** Render a valid page that states no
  subscriptions were processed rather than producing a broken/empty file.
- **Obfuscated run.** See §6 — counts only; no token→real resolution.
- **Very large tenant.** Tier 1 stays O(subs) with no file parsing; Tier 2 is
  O(subs) file opens with a progress bar. The master page itself is small
  (one row per sub + optional charts), not one row per resource.
- **Broken per-sub HTML link.** The link is best-effort; a missing target
  degrades to a disabled/greyed link, not a hard error.

## 12. Testing approach

Consistent with the project's "validate OUTPUT, not internals" philosophy and
the existing Pester layout under `Tests/`:
- Build a synthetic set of 2–3 per-sub `Inventory_*.json` fixtures (no live
  Azure) plus a synthetic `$SubIndex`, invoke `MasterSummary.ps1`, and assert
  on the produced HTML:
  - totals equal the sum of the fixtures,
  - every subscription has exactly one row,
  - failed/empty subs are flagged,
  - an obfuscated fixture set yields the obfuscated privacy banner and leaks no
    real identifiers (reuse the obfuscation-signature assertions from the
    existing obfuscation tests),
  - the file is self-contained (no external `src=`/`href=` to CDNs).
- Add the new tests to the structural set so they run in the scenario matrix
  where applicable. The master is built from artefacts, so most of its testing
  is fixture-based rather than live.

## 13. Open questions

1. **Switch name & default** — `-MasterSummary` on the wrapper, Tier 1 default
   with `-Detailed` for the service aggregate? Or always Tier 1 and a separate
   `-MasterSummaryDetailed`?
2. **Helper sharing** — copy helpers now (isolated) vs. extract into
   `Common.Functions.ps1` (shared, but makes it an output-affecting change to
   `Summary.ps1`). §7 recommends copy-first.
3. **Link target** — link to the per-sub `.html` inside each report folder, or
   to the per-sub `.zip`? HTML is more directly useful; zip is what gets shared
   server-side. Possibly link the HTML and note the zip name.
4. **Naming** — `MasterSummary_<stamp>.html` alongside the per-sub outputs, or
   a fixed `index.html` at the run-output root for "open this first"
   discoverability?

## 14. Suggested phasing

1. **Phase 1 — Tier 1 index.** `Extension/MasterSummary.ps1` with copied
   helpers, wrapper `-MasterSummary` switch, per-sub links, health banners.
   Fixture-based Pester tests. Cheap, high verification value.
2. **Phase 2 — Tier 2 service aggregate.** Add `-Detailed`: parse per-sub JSON,
   accumulate by service, render run-wide donut/bar, wire `Write-RdaProgress`.
3. **Phase 3 (optional) — helper extraction.** If a third consumer appears,
   extract `ConvertTo-HtmlSafe` / `New-DonutChart` / `New-BarChart` into
   `Common.Functions.ps1` as a dedicated, separately-reviewed change.
